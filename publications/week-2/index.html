FOCUS AREA: CLICKJACKING (SLO 3 + 4)

Click-Jacking otherwise known as ‘UI redress attack’ combines both elements of social engineering and Web application to “hijack” clicks and routing them to dangerous domains or applications. The technique is quite simple, the attacker will first need to create a combination of sylesheets, iframes and text boxes to bait the user into believing that they are typing their passwords to a legit site.

OWASP.org presented an excellent example:

For example, imagine an attacker who builds a web site that has a button on it that says “click here for a free iPod”. However, on top of that web page, the attacker has loaded an iframe with your mail account, and lined up exactly the “delete all messages” button directly on top of the “free iPod” button. The victim tries to click on the “free iPod” button but instead actually clicked on the invisible “delete all messages” button. In essence, the attacker has “hijacked” the user’s click, hence the name “Clickjacking”. - www.owasp.org/index.php/Clickjacking

Overall, this vulnerability classifies in the medium catagory as it preys on human error, and can be remediated in two main ways according to OWASP:

  1. Sending the proper Content Security Policy (CSP) frame-ancestors directive response headers that instruct the browser to not allow framing from other domains. (This replaces the older X-Frame-Options HTTP headers.)
  2. Employing defensive code in the UI to ensure that the current frame is the most top level window.

Many sites were hacked this way, including Twitter, Facebook, Paypal and other sites. They are all fixed, of course but this problem is still relevant today as it is classified as a medium/low level vulnerability.
